### 关键代码理解

- `Person p01 = new Person("melon", 18, "male"); Person p02 = new Person("melon", 18, "male");` 后，`p01 == p02` 为引用比较，通常为 `false`；重写 `equals` 后，字段内容相同则可以返回 `true`。

### 关键代码理解

- `Integer i02 = 666` 不是基本类型变量，而是编译器自动把 `666` 装箱成 `Integer` 对象；`int i04 = i02` 则是自动拆箱。`int i05 = i01 * i02` 中两个包装对象会先拆箱为 `int`，再参与乘法。
- `==` 用在 `Integer` 上比较的是两个引用是否指向同一个对象；`equals` 比较的是包装值是否相等。只有在缓存范围内的自动装箱值，`==` 才可能因为对象复用而表现为 `true`，不能把它当成数值比较的通用写法。
- `BigDecimal.divide` 如果除不尽，通常需要明确精度和舍入模式；这里使用 `RoundingMode.CEILING` 会向正无穷方向取整，适合观察舍入策略，但并不等同于四舍五入。
- `String.substring(0, 5)` 使用左闭右开区间 `[start, end)`，会取下标 `0` 到 `4` 的字符；这和数组下标规则能对应起来理解。
- `String[] strings = str01.split(" ")` 得到的是字符串数组，所以后续可以直接用增强 `for` 遍历每一个拆分后的片段。
- `String... strings` 本质上可以按数组处理；调用方可以传多个字符串实参，方法内部用 `for (String s : strings)` 逐个访问。

### 易错点与待修正

- `BigDecimal` 处理小数时建议优先从字符串或整数构造，当前 `BigDecimal.valueOf(10)` 没问题；后续如果使用 `new BigDecimal(0.1)`，会遇到二进制浮点误差问题。

### 关键代码理解

- `Test.Inner inner = test1.new Inner()` 不能直接写成 `new Test.Inner()`，因为 `Inner` 是成员内部类，必须绑定到某个具体的 `Test` 对象；若是静态内部类，创建方式才会变成 `new Outer.Inner()`。
- `System.out.println("成员内部类的name = " + this.name)` 访问的是内部类自己的字段；`Test.this.name` 才明确访问外部类对象的 `name`。这能帮助区分局部变量、内部类字段和外部类字段的优先级。
- `Study1 anonymousStudy = new Study1() { ... }` 与 `Study1 lambdaStudy = () -> ...` 完成的是相近目标，但 Lambda 只能用于函数式接口；`new Person() { ... }` 这种继承普通类的场景不能用 Lambda 替代。
- `Consumer<String>` 的抽象方法是接收一个参数、没有返回值；`BinaryOperator<Integer>` 接收两个同类型参数并返回同类型结果；自定义 `jie_kou_1` 则把这种函数式接口规则落到了自己的接口上。
- `Integer::sum` 能赋值给 `jie_kou_1`，是因为 `Integer.sum(int, int)` 的参数和返回值都能匹配 `sum(int, int)`。
- `throw new TestException(...)` 出现在继承 `Exception` 的异常类型上时，`div2` 必须声明 `throws TestException`，调用 `div2` 的 `main` 也需要继续声明或捕获；而 `StudyException` 和 `NullPointerException` 都属于运行时异常，编译器不强制调用方处理。
- `catch (NullPointerException err)` 捕获了空引用调用 `toString()` 产生的异常，因此 `System.out.println("不受影响的代码")` 仍能执行，这体现了异常处理可以把局部错误限制在受控代码块内。
- `case String s when s.isEmpty()` 必须放在 `case String _` 前面，否则空字符串会先被更宽泛的字符串分支匹配；`test3` 中“大于等于 90”放在“大于等于 80”之前也是同样的从上到下匹配原则。

### 易错点与待修正

- `div2(10, 1)` 当前不会触发异常，主要用于保留编译时异常的方法签名练习；后续建议再用 `div2(10, 0)` 配合 `try-catch` 专门观察自定义编译时异常的捕获流程。
- `test2` 中使用了 `case String _` 和 `case Integer _`，这是较新的 Java 模式匹配语法；如果本地 JDK 或项目语言级别不支持，可能需要开启预览特性或改回传统 `instanceof`/普通 `switch` 写法。
- `jie_kou_1` 接口名使用下划线和小写，不符合 Java 类型命名习惯；后续建议改为 `SumOperation`、`IntBinaryCalculator` 之类的大驼峰命名。
- `String.matches("o+")` 是整串匹配，不是判断是否包含；如果以后想判断包含某段模式，需要继续学习 `Pattern`、`Matcher` 或其他字符串 API。
- `StringBuilder.delete(2, 4)` 仍然是左闭右开区间 `[start, end)`，和 `substring(0, 5)` 的区间规则一致，容易因为把右边界当成包含而删错字符。
- `StudyException` 和 `TestException` 都只传递 message，暂时没有保留 cause；后续学习异常链时，可以补充 `Throwable cause` 构造方法。

### 明天建议

- 给 `div2(10, 0)` 增加 `try-catch` 版本，分别捕获 `TestException` 和通用 `Exception`，观察捕获顺序和异常消息打印；再补一个 `finally`，理解无论是否异常都要执行的收尾逻辑。
- 把 `Test.Inner` 扩展一个静态内部类对照示例，分别写出 `test1.new Inner()` 和 `new Test.StaticInner()` 的创建方式。
- 用同一个函数式接口分别写匿名内部类、Lambda、方法引用三种版本，比较代码长度和适用条件。
- 给 `jie_kou_1` 添加 `@FunctionalInterface`，验证只有一个抽象方法时可以通过编译；再尝试新增第二个抽象方法，观察编译器报错。
- 继续补充 `StringBuilder` 和 `StringBuffer` 的区别，以及正则 `matches`、`replaceAll`、`split` 在字符串处理里的不同用途。

## 本周待巩固

- `public`、`protected`、`private`、默认访问权限在同包、子类、外部类中的差异。
- 构造方法重载、`this(...)` 与 `super(...)` 的调用位置和使用限制。
- `==`、`equals()`、`hashCode()`、`toString()` 的职责区别。
- `Cloneable` 的浅克隆语义，以及为什么 `clone()` 需要处理 `CloneNotSupportedException`。
- 抽象类和接口的适用场景：抽象类偏向“是什么”，接口偏向“能做什么”。
- `enum` 与普通类的相似点和限制，尤其是构造方法私有、常量固定、可带字段。
- `record` 的自动生成方法、不可变数据载体特点，以及它和普通 JavaBean 的差异。
- 包装类自动装箱/拆箱、包装对象比较，以及 `IntegerCache` 对 `==` 结果的影响。
- `BigInteger`、`BigDecimal` 的不可变对象特性、方法式运算和舍入模式选择。
- 数组作为对象、一维/二维数组遍历、数组克隆，以及增强 `for` 的适用边界。
- `String` 常用方法的返回值语义，尤其是截取区间、拆分结果和字符数组转换。
- 密封类 `sealed`、`permits`、`final`、`non-sealed` 的组合规则，需要补一个可编译示例巩固。
- `StringBuilder` 与 `String` 的可变性差异，以及 `delete`、`substring` 这类 API 的左闭右开区间。
- 成员内部类、静态内部类、局部内部类、匿名内部类的创建方式和适用场景。
- Lambda 只能用于函数式接口，匿名内部类仍可用于普通类继承或多抽象方法以外的复杂临时实现。
- 方法引用成立的前提：已有方法的参数列表和返回值要能匹配目标函数式接口。
- 运行时异常与编译时异常的处理要求差异，尤其是 `throws`、`throw` 和后续 `try-catch` 的配合。
- `switch` 模式匹配分支从上到下匹配，带守卫条件的更具体分支应放在宽泛类型分支之前。
